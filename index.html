System Overview

End-to-End Pipeline

Every email captured, every attachment extracted, classified, processed, vectorized, and searchable by Patient ID. Zero documents dropped. Full provenance on every signal.

The system operates as a fully automated pipeline triggered by incoming emails on a dedicated mailbox. Microsoft Graph webhooks detect new messages in real time, extract all attached documents, and route them through a series of Azure AI services — from document classification to clinical entity extraction. Each step produces structured, verifiable outputs rather than AI-generated summaries, ensuring that clinicians can trust every piece of information displayed.

At the end of the pipeline, every clinical signal is stored as a structured card in Cosmos DB, indexed by Patient ID, and made accessible through a Chrome Extension that integrates directly into the EMR workflow. A parallel monitoring dashboard tracks the health of the entire system — emails received, documents processed, errors encountered — with zero tolerance for silent failures. Any document that encounters a processing issue is flagged and queued for human review, never dropped.

01

Email Ingestion & Attachment Extraction

Webhook on dedicated mailbox via Microsoft Graph. Every email captured, attachments extracted, video MIME types detected and flagged. Zero documents dropped — failures create human intervention tasks.

The ingestion layer is built on Microsoft Graph API subscriptions that listen to a dedicated mailbox in real time. When an email arrives, the webhook fires immediately, authenticating via Azure Managed Identity (zero secrets stored anywhere), and extracts every attachment regardless of format — PDFs, DOCX, images, spreadsheets, or any other file type. Each attachment's MIME type is inspected: video attachments (video/*) are detected and flagged with their count and type, while all documents are uploaded to Azure Blob Storage with full metadata (source email ID, timestamp, and Patient ID when parseable from the email subject or body).

The critical design principle here is zero document loss. If any step in the ingestion process fails — whether it's a network timeout, a corrupt attachment, or an unsupported format — the document is still stored in Blob Storage and a human intervention task is created automatically. The system never silently drops a document. Every failure is visible, traceable, and actionable. This error-handling philosophy propagates through the entire pipeline and is surfaced in the monitoring dashboard described in Section 07.

02

Document Classification

Azure AI Document Classification API routes each attachment to the correct extraction model. Multi-report documents detected and split. Unrecognized types flagged for human review.

Once a document is stored in Blob Storage, it is submitted to a custom-trained classifier built on Azure AI Document Intelligence v4.0. This classifier is trained specifically on the client's document taxonomy — lab reports, clinical notes, denial letters, imaging reports, and any other document type relevant to the practice. The classification output includes a confidence score and, critically, a list of sub-documents detected within a single file. This multi-report detection is essential because a single fax or upload often contains multiple concatenated reports from different providers, each with its own title, author, and date.

When a multi-report document is detected, the system splits it into individual report segments using page range information from the classifier output. Each sub-report gets its own title and author extracted, and is processed independently through the rest of the pipeline. Documents that fail classification or fall below the confidence threshold are not discarded — they are flagged as "unrecognized" and routed to the human review queue, maintaining the zero-drop guarantee established in the ingestion layer.

03

Content Extraction — Form Recognizer v4

Custom extraction models per document type. Structured fields, bounding box polygons for visual receipts, date segregation (report vs visit date), embedded video detection.

Each classified document is routed to a custom extraction model trained for its specific type. A lab report model extracts different fields than a denial letter model — this per-type specialization is what delivers high accuracy on messy, unstructured faxes. Form Recognizer v4 returns structured fields (title, author, patient DOB, clinical findings) along with polygon coordinates for every extracted value. These bounding box polygons are the foundation of the Visual Receipt system: they pinpoint exactly where on the original document each piece of data was found, enabling one-click source verification by clinicians.

Date segregation is handled as a distinct post-processing step. Medical documents typically contain multiple dates with different meanings — report date, visit date, admission date, discharge date — and conflating them leads to clinical errors. The extraction module parses all date fields and categorizes them by context, ensuring that each signal card carries the correct date type. Additionally, documents are scanned for embedded video content (PDF RichMedia annotations, DOCX OLE objects, PPTX media slots) — detected videos are flagged with their count and location but not indexed, as per the project scope.

04

Healthcare Entity Recognition & Signal Cards

Text Analytics for Health extracts clinical entities. Each entity becomes a structured signal card — non-AI-authored, verifiable extraction with source tagging.

The extracted text from each document is submitted to Azure AI Language's Text Analytics for Health, which performs named entity recognition (NER) specifically tuned for clinical vocabulary. The service identifies diagnoses (linked to ICD-10 and SNOMED CT codes), medications (with dosage and frequency), symptoms and physical findings, examination names, and treatment procedures. Beyond individual entities, it also extracts relationships between them — which medication is prescribed for which condition, what dosage is associated with what drug, which finding was observed at what time.

Each extracted entity is assembled into a Signal Card using Pydantic models for strict schema validation. A Signal Card is not an AI summary — it is a structured, non-authored extraction that contains the exact entity text, its clinical category, confidence score, UMLS/ICD-10/SNOMED CT code links, and a source tag pointing to the exact document name, page number, and character offset where the entity was found. The accompanying Visual Receipt (a cropped, highlighted snippet of the original document page) provides instant visual verification. This design ensures that every piece of clinical information displayed to a clinician is traceable back to its source with one click.

05

Chunking, Vectorization & Cosmos DB

Documents split into semantic chunks, vectorized via Azure OpenAI Embeddings, stored in Cosmos DB with vector index. Signal cards linked. Everything indexed by Patient ID.

In parallel with signal card generation, each document's full extracted text is split into semantic chunks of approximately 500 tokens with a 50-token overlap. The chunking algorithm respects paragraph and sentence boundaries to preserve contextual coherence. Each chunk is then vectorized using Azure OpenAI's text-embedding-3-large model, producing a 3072-dimension vector representation that captures the semantic meaning of the text. This enables clinicians to search across a patient's entire document history using natural language queries rather than exact keyword matching.

All data — vector chunks, signal cards, document metadata, and processing status — is stored in Azure Cosmos DB using the NoSQL API with DiskANN vector indexing. The database is partitioned by Patient ID, which means all data for a given patient is co-located for fast retrieval. Each signal card maintains cross-references to its source chunks, and each chunk carries a reference back to its source document and page. This bidirectional linking enables the Chrome Extension to seamlessly navigate from a search result to a signal card to the original document page, creating a complete provenance chain.

06

Chrome Extension, Patient ID Sync & Search

Chrome Extension captures Patient ID from EMR URL, queries Cosmos DB via Flask API. Displays signal cards and enables semantic search. Click any card to verify in the original document.

The Chrome Extension is the primary interface for clinicians. It operates as a lightweight side panel that activates when the user navigates to a patient page in the EMR system. The extension parses the EMR URL to extract the Patient ID, then calls the Flask Blueprint API to retrieve all signal cards, documents, and search capabilities scoped to that specific patient. The JavaScript in the extension is intentionally minimal — it handles URL detection, API calls via fetch, and DOM rendering only. All business logic, data processing, and security enforcement lives in the Python backend.

Three API endpoints power the extension. The cards endpoint returns all signal cards for the patient, each with its entity type, value, confidence, and source reference. The search endpoint accepts natural language queries (e.g., "latest blood work results" or "denial rationale for MRI"), embeds them using the same OpenAI model, and performs a vector similarity search scoped to the patient's documents. The documents endpoint lists all files associated with the patient, their processing status, and provides short-lived SAS URLs for direct document viewing. When a clinician clicks on any signal card, the source verification flow opens the original document at the exact page with the relevant region highlighted — providing the verifiable "Visual Receipt" that is the core success criterion of this project.

07

Dashboard Monitoring & Error Handling

Tracks emails received, documents processed, error counts, per-document status. Zero documents dropped — problems flagged and queued for human intervention.

The monitoring dashboard provides a real-time operational view of the entire pipeline. It displays four primary metrics — total emails received, documents successfully processed, current error count, and documents pending human review — updated in real time via Cosmos DB Change Feed. Below the summary counters, a paginated document list shows every file in the system with its current status (processed, error, pending review), the pipeline stage it reached before any failure, the specific error detail if applicable, and a timestamp for each state change. This gives operations staff immediate visibility into the system's health without digging through logs.

The error handling philosophy is proactive, not reactive. When a document fails at any stage — classification, extraction, NER, or vectorization — it is not retried silently. Instead, a human intervention task is created with the full context: which document, which stage, what error, and a direct link to the original file in Blob Storage. A staff member can review the issue, decide whether to re-queue the document for processing (after fixing the root cause) or annotate it with notes for later resolution. The goal is zero documents in error state — the dashboard makes this goal visible and actionable at all times, with Azure Monitor Application Insights providing the underlying telemetry, exception tracking, and alerting infrastructure.

Delivery Plan

Milestones & Evidence

Each milestone has defined deliverables and acceptance criteria with verifiable evidence.

The project is structured into five consecutive milestones, each with a clear scope boundary and measurable acceptance criteria. This phased approach ensures that the client sees working, testable output at regular intervals rather than waiting for a single big-bang delivery. Each milestone builds on the previous one — infrastructure must be solid before classification can work, extraction must be proven before NER can run, and so on. This dependency chain is by design: it catches integration issues early and prevents expensive rework later in the project.

Every milestone concludes with a formal acceptance review. The deliverables are demonstrated live (screen-recorded as per the Recorded Evidence Work Policy), and the client validates each acceptance criterion against the evidence provided. Payment for the milestone is released only after this review is complete and the client confirms satisfaction. If a criterion is not met, the issue is documented and resolved before the milestone is closed. This structure protects both sides: the client gets verified quality at each step, and the developer has clear, unambiguous completion criteria.

DevOps

CI/CD — GitHub Actions

Trunk-based development. Every push triggers lint + test. Merge to main triggers staging deploy. Tag triggers production with blue-green rollout.

The CI/CD pipeline is built on GitHub Actions with four distinct workflow files, each triggered by different events. The CI workflow runs on every push to any branch and every pull request to main — it installs dependencies, runs ruff for Python linting, executes the full pytest suite (unit + integration tests with a coverage gate of 85%), and performs a bandit security scan. No code reaches the main branch without passing all these gates. This automated quality enforcement eliminates the "it works on my machine" problem and ensures that the main branch is always in a deployable state.

Deployment follows a staged promotion model. When code is merged to main, the staging workflow automatically builds a Docker image, scans it with trivy for container vulnerabilities, pushes it to Azure Container Registry, and deploys it to the staging Container App environment where smoke tests run. Production deployment is triggered only by tagging a commit on main (e.g., v1.0.0), which promotes the staging-tested image to the production Container App using a blue-green strategy with traffic splitting. If the health check fails after deployment, an automatic rollback restores the previous version. A separate infrastructure workflow watches for changes in the /infra directory and applies Bicep templates through a validate → dry-run → manual approval → deploy sequence.

Engagement Model

Pricing — Hour-Range Milestones

Transparent hour-range model per milestone. Every billed hour backed by screen-recorded evidence. You pay only for documented, verifiable work.

This engagement uses an hour-range model rather than a fixed price, and there is a deliberate reason for this. The project involves custom AI model training where the number of iterations required depends on factors that cannot be predicted upfront — the quality and consistency of the client's real documents, the number of distinct document types that need custom models, and the complexity of multi-report compounds. A fixed price would force either inflated padding (overcharging) or corner-cutting to stay in budget (underdelivering). The hour-range model eliminates both problems by aligning cost with actual verified effort.

The Recorded Evidence Work Policy is the mechanism that makes this model trustworthy. Every working session is screen-recorded — IDE, terminal, Azure portal, browser testing, everything. Each recording is timestamped, mapped to a specific GitHub issue or task, and tied to the corresponding Git commits. The client has permanent access to all recordings via a shared cloud folder. Weekly invoices include direct links to the relevant recordings for that billing period. If any hour is disputed, the client reviews the recording and the hour is not billed unless the work is clearly visible. This creates a fully transparent, audit-ready engagement where every euro spent is documented.

Hosting

Publication — Azure Static Web Apps

This architecture document is published as a static site via Azure Static Web Apps, connected to the project's GitHub repository for automatic deployments.

Azure Static Web Apps is chosen for hosting this documentation because it provides a globally distributed CDN, automatic HTTPS with free SSL certificates, and a seamless integration with GitHub Actions — all on the free tier. When this HTML file is pushed to the project's GitHub repository, a GitHub Action workflow (auto-generated by Azure during the initial setup) builds and deploys the static site within approximately 60 seconds. This means that any update to the architecture document is live immediately after a push to main, keeping the documentation in sync with the actual codebase.

The deployment is entirely infrastructure-as-code. The Azure Static Web App resource can be provisioned with a single Azure CLI command that links it to the GitHub repository. No manual configuration is required — Azure handles the webhook setup, the build pipeline, and the CDN distribution automatically. For production use, a custom domain can be attached with a single CLI command, and Azure provisions and manages the SSL certificate. This approach ensures that the documentation site has the same level of automation, reliability, and auditability as the application itself.
